== Ingress ==

  Ingress is enabled (type: ingress). The chart provisions an Ingress resource
  for the API server at:
{{- range .Values.apiServer.ingress.hosts }}
    - https://{{ .host }}{{ range .paths }}{{ .path }}{{ end }}
{{- end }}
{{- if .Values.apiServer.ingress.tls }}

  TLS is configured. cert-manager will issue a certificate for the host(s)
  above via the issuer referenced in the ingress annotations
  (cert-manager.io/issuer). If the certificate is not ready, check:

    kubectl describe certificate -n {{ .Release.Namespace }}
    kubectl describe certificaterequest -n {{ .Release.Namespace }}
{{- end }}

  The ingress controller (e.g. ingress-nginx) must be installed in the cluster:

    helm install ingress-nginx ingress-nginx/ingress-nginx

  Quick test with a wildcard DNS (no real domain needed):
    # 1. Get the ingress controller LB IP
    kubectl get svc -n ingress-nginx ingress-nginx-controller \
      -o jsonpath='{.status.loadBalancer.ingress[0].ip}'
    # 2. Use nip.io to resolve a hostname to that IP
    curl -k https://paprika.<LB_IP>.nip.io/healthz
{{- else }}

== Upgrade ==

  helm upgrade {{ .Release.Name }} . -n {{ .Release.Namespace }} \
    --values my-values.yaml

CRDs are upgraded automatically with the chart. On schema changes, existing
resources are converted by the CRD storage version migration (handled by
the API server — no manual intervention needed).{{ if .Values.velero.enabled }}

== Backup ==

Velero backup schedule is active. To trigger an immediate backup:

  velero backup create --from-schedule {{ include "paprika.resourceName" (dict "suffix" "backup" "context" $) }} -n {{ .Values.velero.namespace }}

To restore:

  velero restore create --from-backup <backup-name> --namespace-mappings {{ .Release.Namespace }}:<new-ns>

See https://velero.io/docs/ for detailed restore options.{{ end }}

Paprika consolidates the tools your team needs into a single operator — no more stitching together Argo projects with different UIs and upgrade cycles.

Unified Resource Model

Replace ArgoCD, Rollouts, and Workflows with a single Application CRD. One operator, one resource model, one consistent API. No more stitching together three projects with different conventions.

Automated rollbacks

Progressive Delivery

Canary and blue-green deployments with configurable step weights. Traffic shifts gradually with health verification at every step. Roll back automatically if error rates spike.

Istio + Gateway API

Pluggable Traffic Routing

Built-in support for Istio VirtualServices and Gateway API HTTPRoutes. Per-stage traffic router configuration gives you fine-grained control over how traffic reaches your canaries.

Any cluster, per stage

Multi-Cluster by Default

Stage-level cluster references let you deploy to different clusters per environment. No pre-configured cluster secrets required. Each stage can target a different Kubernetes cluster.

Verify before promote

CEL Health Verification

Health checks evaluate every resource at every stage using Common Expression Language. Built-in resource health rules, approval gates, and automated rollback on failure.

How It Works

From Source to Production in Three Steps

Define, promote, verify — Paprika handles rendering, traffic routing, and health checking so you don't have to.

Step 1

Define Your Application

Create a single Application resource that references your Helm chart, Git repo, or S3 bucket. Specify build steps, stages, and deployment strategy in one manifest. Paprika handles the rest.

apiVersion: pipelines.paprika.io/v1alpha1
kind: Application
metadata:
  name: my-app
spec:
  source:
    git:
      url: https://github.com/example/app
      revision: main
  strategy: Canary
Step 2

Promote Through Stages

Paprika renders manifests using the Helm SDK, runs pipeline steps as Kubernetes Jobs, and promotes releases through stages. Each stage supports its own canary config, traffic routing, and cluster target.

  stages:
    - name: staging
      canary:
        steps:
          - weight: 10
          - weight: 50
          - weight: 100
    - name: production
      gates:
        - name: approve
          type: ManualApproval
Step 3

Verify and Release

Health checks evaluate every resource at each stage. Traffic routing shifts weights gradually. Approval gates pause for manual verification. If checks fail, Paprika rolls back automatically.

  stages:
    - name: production
      trafficRouter:
        istio:
          virtualService: my-app
          host: app.example.com
